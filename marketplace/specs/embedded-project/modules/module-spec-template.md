# 模块名称规范

## 1. 目标

说明该模块解决什么问题，以及它在项目分层中的位置。

## 2. 所属层级

必须明确该模块属于：

- `app/`
- `services/`
- `protocols/`
- `middleware/`
- `drivers/`
- `bsp/`
- `platform/`
- `vendor/`
- `common/`
- `config/`

## 3. 适用范围

说明该规范适用于哪些目录、文件和接口。

## 4. 模块职责

- MUST: 说明该模块允许做什么。
- MUST: 职责必须符合所属层级。

## 5. 禁止职责

- MUST: 说明该模块禁止做什么。
- MUST NOT: 不允许模块承担上层或下层职责。

## 6. Public API 规则

说明 public header 应暴露哪些内容，不应暴露哪些内容。

- MUST: public API 必须说明参数、返回值、错误码和调用上下文。
- MUST NOT: public API 不得暴露无保护内部状态。

## 7. 状态和数据所有权

说明模块内部状态、共享数据、context、buffer 的所有权。

## 8. 并发和 ISR 规则

说明是否允许 ISR 调用，是否涉及 task、mutex、queue、event flag。

## 9. 错误处理和日志

说明错误码、日志、诊断数据。

## 10. 测试要求

说明如何单元测试、mock 或最小验证。

## 11. 检查清单

- [ ] 是否明确所属层级？
- [ ] 是否遵守依赖方向？
- [ ] 是否没有越层访问？
- [ ] 是否没有把业务逻辑放进 driver？
- [ ] 是否没有把硬件访问放进 `app/`？
- [ ] 是否有明确 public API？
- [ ] 是否有错误处理路径？
- [ ] 是否有测试或验证策略？
