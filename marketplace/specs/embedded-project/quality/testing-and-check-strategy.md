# 测试和检查策略

## 可测试设计

- MUST: 纯逻辑模块必须尽量脱离硬件运行。
- MUST: CRC、ring buffer、协议解析、状态机、参数校验等模块必须优先支持单元测试。
- MUST: 硬件相关模块应通过 mock、fake 或 adapter 隔离硬件依赖。

## 最小验证

- MUST: 每次新增模块至少提供编译级验证。
- MUST: 每次修改协议、状态机、公共工具、错误码时必须说明验证方式。
- MUST: 修改 public API 时必须检查调用方。
- SHOULD: 每个模块提供最小 usage 示例或测试入口。

## Agent 自检

- MUST: agent 修改代码后必须输出自检结果。
- MUST: 自检结果必须说明修改了哪些文件、属于哪一层、是否符合依赖方向、是否存在未验证内容。
- MUST NOT: agent 不得声称测试通过，除非实际执行了测试或构建命令。

## 检查顺序

1. 检查目录和层级是否正确。
2. 检查 public API 是否清晰。
3. 检查依赖方向是否单向。
4. 检查错误处理和超时路径。
5. 检查并发、ISR、DMA 影响。
6. 执行可用的构建、测试或静态检查。
